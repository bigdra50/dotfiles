# Writing Style Rules

## AI臭さ除去

### 避けるべき表現

冗長な修飾語・誇張表現・定型句を避ける。

## 自己完結する文書

文書はそれ単体で読んで意味が通る「自己完結」を基本とする。
書き手と文脈を共有しない第三者（未来の自分を含む）の視点で書く。

- 執筆時の時系列・作業経緯を本文に残さない。「新仕様/旧仕様」「N回目」「先ほど作成した」「FB対応後に更新」など、書き手と特定の読者しか知らない文脈は書かない
- 文書を更新するときは過去の版との差分や経緯を書き足さず、常に「現時点の完成形」として書き直す
- 変更の履歴が必要なら本文ではなくバージョン管理（git）に委ねる
- 例外: 別文書を参照すると読み手に有用なときは、Markdownリンク `[表示名](path)` で相互参照してよい。「別ファイルに全文がある」等の案内は、執筆時系列ではなく読み手の動線として書く

コミット・PRでの同種の規律は commit.md「ローカル文脈の排除」を参照。

## 機械生成テキストの流し込み禁止

エージェント・パイプラインの中間出力（分析結果、検証エージェントの判定文、JSONから組み立てた文章）を、人間が読む文書（Issue、PR、docs、notes、チャット報告）へ無加工で貼らない。
中間出力はエージェント間の受け渡しに最適化されており、括弧註が多重に入る長文一段落など、人間の読み手には不適。
書いたのが誰（どのモデル）でも、読み手が一読で構造を掴めなければ品質不足とみなす。

- 投稿前に必ず「短い文・箇条書き・見出し」へ再構成する。1文1主張に分解し、括弧註への圧縮をほどく
- 検証ログ・判定文の全文を `<details>` に退避して済ませない。読み手に必要な要点だけを箇条書きに抽出する
- 数値・file:line・固有名詞などの事実情報は再構成で落とさない。落とすのは冗長な言い回しであって情報ではない

### GitHub Issue 本文の構成

- 「問題 / 対応（チェックリスト）/ 補足（確定した事実）」の3部構成を基本にする
- 対応はチェックリストにし、済み・残りが一目で分かる状態を保つ
- 進行中のissueは経緯を追記で積まず、本文を常に現時点の完成形へ更新する（経緯はコメントとgitに委ねる）

## 改行

- 句点（。）またはピリオド（.）の後で改行する（一文一行）
- 箇条書きの各項目も1行にする
- 段落の区切りは空行で表現する
- trailing spacesは入れない（soft line break）
- レンダリング上の改行制御はプラットフォーム固有設定で対処する
