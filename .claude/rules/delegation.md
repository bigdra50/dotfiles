# AI委譲ルール

各LLMの得意領域とコストを踏まえて作業を分担する。

## 基本方針: 実装委譲は Cursor CLI（grok）を優先

Cursor は自社提供モデルを安価に提供している。
性能が同等なら、実装タスクは Claude 系サブエージェントではなく cursor-agent（grok）へ委譲し、Claude の消費をオーケストレーション（仕様書き・レビュー・検証・commit）に絞る。

| 役割 | 委譲先 | モデル | 特性 |
| --- | --- | --- | --- |
| 実装（第一候補） | `cursor-agent` | `grok-4.5-fast-xhigh` | 安価・高速（1 タスク 1〜5 分）。内部で composer-2.5 に実装委譲し自己レビューする。タイトな仕様への忠実度が高い |
| 探索・分析・仕様未確定 | Claude 系サブエージェント | sonnet 等 | 自走してコードを裏取りできる。曖昧な仕様の訂正力は grok より上 |
| レビュー・議論 | `copilot` | GPT 系 | read-only の設計分析・裏取り用。実装には使わない |
| 統括 | Claude Code（自分） | - | 仕様書き・事実の裏取り・diff レビュー・独立検証・commit/push・破壊的操作 |

## grok 委譲プロトコル（実証済み）

- 起動形: `agent -p --force --trust --model grok-4.5-fast-xhigh "<仕様>"`
- 仕様には「検証済みの背景事実・変更内容・検証コマンド・変更ファイル制約・commit 禁止」を明記する
- **ブリーフィングの事実は渡す前に自分で裏取りする**。grok は与えられた背景を独立に検証せず、誤った前提をそのまま成果物へ反映する
- 検証の分担: grok には高速なターゲットテストのみ実行させ、全スイートは Claude 側で回す（agent セッション内の長い検証はタイムアウトの元）
- 破壊的操作（rm・鍵の削除等）は委譲せず Claude が直接実行する
- 完了後は Claude が diff レビュー → 独立検証 → commit の順で締める

## 長時間タスクの監視

Bash 同期実行の 10 分上限は CLI の制限ではない。
長時間タスクは backgroundで起動して時間制限を外し、進捗を監視する。

- `--output-format stream-json` でツール呼び出し単位のイベントをログへ流す（途中経過が読める）
- ログ停滞（例: 5 分間サイズ不変）を検知するウォッチャを併走させ、スタックと無報告完了を検出する

## CLI 存在確認

マシンごとにインストール状況が異なる。
委譲前に `command -v` で確認し、不在なら自分で実行する。

| 委譲先 | 確認コマンド |
| --- | --- |
| `cursor-agent` | `command -v agent` |
| `copilot` | `command -v copilot` |
| `codex` | `command -v codex` |

## いつ自分で実装するか

- 1〜2 ファイル・数行の修正（委譲のオーバーヘッドが勝る）
- 試行錯誤が必要 / 設計判断が未解決のタスク
- ユーザーが直接やらせたいと明示
- 委譲先が未インストール・レート制限・認証エラー

## 安全規則

### 返却フォーマット

サブエージェントに以下で返却させる:

- 要約（1〜3 行）、変更ファイルパス一覧、未解決の質問・トレードオフ、`file:line` 引用

### 破壊的 git 禁止

委譲プロンプトに必ず含める:

- `git checkout` / `git restore` / `git stash` / `git reset` 禁止
- 許可は `git mv` と読み取り系のみ
- 委譲後に `git status --short` で dirty ファイルの生存を確認

過去に委譲先が `git checkout -- <file>` でユーザーの未コミット編集を破壊した事故あり。

## 重要な判断はLLM間で議論する

議論の余地があり重要な判断は、ユーザーに問う前に copilot（GPT 系）と議論し結論を提示する。

- 決着がつくまで繰り返す（1 ターンで終えない）
- reasoning effort: 通常 `high`、複雑な判断 `xhigh`
- 既定 copilot → 不可時 codex へフォールバック（1 議論あたり 1 回まで。切替後は切替先で続行）
- 両方不可なら打ち切り、自分の見解を提示

## セッションの継続

同じ相手との継続は `--continue`（直近）または `--resume <id>`（特定セッション）を使う。
