## 開発スタイル

TDD で開発する（探索 → Red → Green → Refactoring）。
KPI やカバレッジ目標が与えられたら、達成するまで試行する。

## コード設計

- コントラクト層（API/型）を厳密に定義し、実装層は再生成可能に保つ
- 静的検査可能なルールはプロンプトではなく linter か ast-grep で記述する

## リポジトリ管理（ghq）

ローカルリポジトリは `ghq` で管理している。

```bash
ghq root                   # リポジトリルート取得
ghq list                   # ローカルリポジトリ一覧
ghq list | rg <keyword>    # リポジトリ検索
ghq get <repo>             # clone
```

別リポジトリの参照やcloneが必要な場合は `ghq` を使う。

## Output Style

### 作業完了時のASCII図解

実装完了・作業完了後、変更内容をASCII図で視覚的に示す。

作業内容に応じて適切な図を選択：

| 作業内容              | 図の種類         |
| --------------------- | ---------------- |
| API・処理フロー実装   | シーケンス図     |
| クラス・型の追加/変更 | クラス図         |
| モジュール構成変更    | コンポーネント図 |
| 状態管理実装          | 状態遷移図       |
| データ構造変更        | ER図             |
| パイプライン・CI/CD   | フロー図         |

図解のルール：

- ASCII文字のみ使用（`+`, `-`, `|`, `>`, `*` 等）
- 変更箇所が明確にわかること
- 過度に巨大な図は避け、要点に絞る

### ランダムASCIIアート

作業完了時、低確率（体感5回に1回程度）で遊び心のあるASCIIアートを添える。

方針：

- 図解の代わりではなく、図解に加えてオマケとして出す
- CLIカルチャーのミーム、名言、小ネタなど自由に
- 短く、3〜6行程度に収める
- 作業内容や状況に合ったものを選ぶとなお良い

### 専門用語・略語の初出説明

ユーザーが普段使わない略語・専門用語をセッションで初めて使うときは、その直後に概要を 1 行添える。

- 同一セッション内の 2 回目以降は不要
- ユーザーの専門分野で自明な語には付けない（冗長回避）
